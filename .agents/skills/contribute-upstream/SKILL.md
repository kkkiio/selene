# Contribute Upstream

Contribute fixes from a long-lived fork without leaking fork-only changes. Keep the issue easy to scan and the PR independently reviewable.

## Respect the requested action

- For an assessment, inspect and report without writing to GitHub or changing repository history.
- For an issue request, create the issue after reproducing the problem.
- For a PR request, create a branch, commit, push to the fork, and open the upstream PR.
- Treat issue and PR creation as separate write actions; perform only those the user requested.
- Use the installed GitHub connector for issue and PR operations. Follow the applicable GitHub publishing skill for authentication, branch naming, draft state, and push mechanics.

## Resolve the repository relationship

1. Inspect remotes, default branches, the current checkout state, and repository instructions.
2. Identify the writable fork remote and the canonical upstream remote from URLs; do not infer them from remote names alone.
3. Fetch the upstream default branch before comparing code.
4. Compare upstream with the fork commit containing the candidate fix. Separate upstreamable behavior from fork namespace, packaging, release, dependency, documentation, and formatting changes.
5. Stop and ask when the canonical upstream or writable fork is ambiguous.

Never push a contribution branch to the canonical upstream remote. Push it to the user's fork and target upstream from the PR.

For an assessment, keep the upstream default branch as a remote-tracking ref and use `git show`, `git diff`, and similar read-only commands. Do not create, switch, or delete branches merely to inspect upstream. For a PR, create a contribution branch from the fetched upstream default branch only after confirming the current checkout can switch safely. Stop rather than stash, reset, clean, overwrite, or discard user changes.

## Prove the upstream problem

1. Inspect the exact implementation on the fetched upstream branch.
2. Compare the affected source, dependencies, and build configuration with the current checkout. When they are equivalent, a reproducer run in the current checkout is valid evidence for that upstream commit; record the equivalence explicitly.
3. For an assessment, prefer existing tests and non-persistent diagnostics. If deterministic reproduction requires changing the checkout, report the strongest read-only evidence and remaining uncertainty instead of manufacturing temporary repository state.
4. For an issue or PR, add only a minimal regression test after confirming local edits are authorized and isolated from unrelated user changes.
5. Run the narrowest relevant test and capture the expected/actual difference.
6. Apply the candidate fix when requested and rerun the same test.
7. Confirm the test fails without the fix and passes with it before publishing a PR.

Do not report a confirmed upstream bug from code similarity alone. If a deterministic test is impractical, state the strongest available evidence and the remaining uncertainty.

## Write the issue

Search open and closed upstream issues for duplicates before creating one.

Keep the issue short and symptom-led:

- Title the observable failure.
- State what happens and why it matters in one short paragraph.
- Include a minimal, runnable reproducer.
- Show expected and actual results compactly, using a small table when useful.
- Include the tested upstream commit and only relevant environment details.
- Omit suspected root cause, code-level analysis, workaround, and proposed fix. Put implementation details in the PR.

Avoid greetings, generic background, oversized templates, and claims that the report was generated by an AI.

## Build the upstream patch

1. Continue on the contribution branch used to prove the problem.
2. Keep the smallest complete behavior fix and focused regression coverage.
3. Update the upstream changelog only when repository instructions require it.
4. Review the full diff against upstream, including generated and formatted changes.
5. Remove fork-only metadata, roadmap documents, unrelated cleanup, dependency churn, and drive-by formatting.
6. Keep the fix in one upstreamable commit unless the upstream repository requires another structure.

Do not seed the contribution branch with the fork's complete diff. Apply only the selected upstreamable changes to the branch created from upstream.

## Validate and publish

1. Run the focused regression test with the repository's warning policy.
2. Run the smallest broader check that covers the changed package or module.
3. Attempt repository-wide checks when proportionate. Record unrelated baseline failures without expanding the PR to fix them.
4. Stage only named contribution files and inspect the staged diff.
5. Commit with a terse, natural message that follows repository conventions.
6. Push the branch to the writable fork remote.
7. Open the PR against the canonical upstream default branch.

Write a concise PR body containing:

- a two-item summary at most;
- the root cause and developer/user impact in one paragraph;
- `Fixes #<issue>` when an issue exists;
- the exact passing checks.

Respect an explicit ready or draft request. Otherwise follow the active GitHub publishing workflow's default. Allow upstream maintainers to modify the branch when supported.

## Finish

Return links to the issue and PR, the contribution branch and commit, passing checks, any unrelated validation blocker, and the current checkout state.
